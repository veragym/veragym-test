⚙️ 서비스 워커 캐싱 전략
N
HTML / JS 파일 요청
Network-First 전략
→ 항상 최신 버전 제공, 실패 시 캐시 폴백
C
아이콘 / 매니페스트 요청
Cache-First 전략
→ 캐시 우선, 없으면 네트워크
U
앱 업데이트
GitHub push → 앱 종료 후 재실행 시 자동 반영
버전 범프 불필요
4. 보안 · RLS 정책 요약
테이블 / 버킷 역할 SELECT INSERT UPDATE DELETE 조건
trainers authenticated ✓ ✗ ✗ ✗ 본인 id만
members authenticated ✓ ✓ ✓ ✗ trainer_id = 본인
workout_logs authenticated ✓ ✓ ✓ ✗ trainer_id = 본인
workout_logs anon ✓ ✗ ✗ ✗ token 일치 회원 소유
exercise_refs all ✓ ✗ ✗ ✗ 전체 공개 읽기
trainer_exercises authenticated ✓ ✓ ✓ ✓ trainer_id = 본인
session-photos (storage) anon ✓ ✗ ✗ ✗ storage_is_active_member() 함수 검증
session-photos (storage) authenticated ✓ ✓ ✗ ✗ 트레이너 업로드 허용
⚡ SECURITY DEFINER 함수: storage_is_active_member(p_member_id TEXT) — anon이 RLS 보호된 members 테이블을 참조할 수 있도록 postgres 권한으로 실행. 향후 추가되는 모든 회원에 자동 적용.
5. PWA 구성
📱 설치 지원
▸iOS Safari: 공유 → 홈 화면에 추가
▸Android Chrome: 설치 프롬프트 또는 메뉴
▸3개 앱 각각 독립 manifest.json 보유
▸PWA 아이콘: icon-192.png, icon-512.png
🔄 업데이트 정책
▸HTML/JS: network-first → 자동 최신화
▸아이콘/매니페스트: cache-first
▸앱 완전 종료 후 재실행 시 반영
▸사용자 수동 캐시 삭제 불필요
⚠️ bfcache 대응
▸iOS PWA 탭 이동 시 JS 상태 freeze 현상
▸pageshow 이벤트로 bfcache 복원 감지
▸_initRunning 플래그 리셋 후 재초기화
▸적용 파일: trainer-dash.html, exercise-library.html
🏗️ 호스팅
▸GitHub Pages: veragym.github.io/veragym-app
▸main 브랜치 push → 자동 배포
▸별도 빌드 과정 없음 (순수 HTML/JS)
▸config.js: Supabase URL/Key 관리
